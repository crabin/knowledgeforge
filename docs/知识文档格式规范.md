# KnowledgeForge 知识文档格式规范

> 目的：为所有需要保存到本地知识库的知识文档提供统一、通用、易读的 Markdown 存储结构，确保文档既适合人工阅读，也适合后续结构化抽取、Neo4j 路径关联、质量检测和版本管理。
>
> 当前代码额外要求：知识点文件必须包含 KnowledgeForge contract 块，用于记录证据占位、文件级链接查询任务和完成状态；该 contract 是领域级链接队列和 Neo4j 节点状态同步的数据合同。默认产物是“架构文档”，完整知识文章通过后置补全文档动作生成。

## 1. 适用范围

本规范适用于 KnowledgeForge 中所有进入知识沉淀链路的 Markdown 文件，包括：

- 框架证据文档
- 原始资料整理文档
- Agent 中间产物文档
- 结构化后的知识文章
- 子领域补充材料
- 观点、趋势、案例类汇总文档

领域结构目录索引文档和版本记录文档可以使用专门模板，但仍应遵守基础元信息、来源追溯和路径稳定性要求。

---

## 2. 文件命名与保存位置

### 2.1 保存位置

知识文档按以下层级保存：

```text
save/{领域名称}/{子领域名称}/{文档文件名}.md
```

领域目录下必须包含结构目录索引文档：

```text
save/{领域名称}/README.md
```

### 2.2 文件命名

推荐文件名格式：

```text
{YYYYMMDD}-{短标题}-{来源类型}.md
```

示例：

```text
20260423-langgraph-state-management-query.md
20260423-agent-workflow-practices-media.md
20260423-local-knowledge-overview-insight.md
```

命名要求：

- 使用小写英文、数字和短横线，减少跨系统路径兼容问题。
- 文件名应稳定，不应因为标题轻微修改频繁变化。
- 同一子领域内文件名不得重复。
- 文件移动或重命名后，必须同步更新 Neo4j 路径关联和版本记录。

## 3. 架构文档结构

默认 `completion_mode=framework` 时，每个知识点文件用于保存领域整体架构、学习定位和证据链接入口，不要求生成完整正文。文件只在 Neo4j 图谱完成两轮架构 review 且通过后生成。

```markdown
---
id: "{stable_document_id}"
title: "{知识节点标题}"
domain: "{领域名称}"
subdomain: "{子领域名称}"
doc_type: "note"
source_type: "query"
agent: "KnowledgeForge"
round: 1
status: "draft"
created_at: "YYYY-MM-DDTHH:mm:ss+09:00"
updated_at: "YYYY-MM-DDTHH:mm:ss+09:00"
version: "v1"
path: "save/{领域名称}/{子领域名称}/{文档文件名}.md"
tags:
  - "{tag}"
sources:
  - title: "Knowledge blueprint scaffold"
    url: "local://knowledge-blueprint"
    publisher: "KnowledgeForge"
    retrieved_at: "YYYY-MM-DDTHH:mm:ss+09:00"
    reliability: "unknown"
---

# {知识节点标题}

## 知识定位

- 领域：{领域名称}
- 子领域：{子领域名称}
- 作用：说明该知识节点在整体知识框架中的位置、边界和证据入口。

## 学习角色与路径

- 学习角色：{领域总览/主题单元/具体知识点/导航索引}
- 建议顺序：{学习顺序或图谱位置}
- 前置知识：{前置节点；没有则写“暂无”。}

## 知识关系

- 上级节点：{上级节点}
- 相关节点：{相关节点}

## 证据与来源

| 编号 | 来源 | 关键信息 | 可信度 | 备注 |
|---|---|---|---|---|
| S1 | {官方文档/标准/权威论文/项目主页} | {支撑的定义、关系或学习路径} | high | {说明} |
```

## 4. 完整知识文档结构

`completion_mode=full_document` 时，系统在架构文档和可信链接完成后再生成完整知识库文档。每篇完整知识文档必须使用以下结构：

```markdown

subdomain: "{子领域名称}"
doc_type: "source|article|summary|case|trend|note"
source_type: "insight|query|media|manual|mixed"
agent: "{agent_name}"
round: 1
status: "draft|reviewed|verified|deprecated"
created_at: "YYYY-MM-DDTHH:mm:ss+09:00"
updated_at: "YYYY-MM-DDTHH:mm:ss+09:00"
version: "v1"
path: "save/{领域名称}/{子领域名称}/{文档文件名}.md"
tags:
  - "{tag}"
sources:
  - title: "{来源标题}"
```

## 5. 字段说明

### 5.1 必填元信息

| 字段 | 说明 |
|---|---|
| `id` | 稳定文档 ID，用于去重、版本追踪和图谱映射 |
| `title` | 文档标题，应清晰表达主题 |
| `domain` | 所属领域 |
| `subdomain` | 所属子领域 |
| `doc_type` | 文档类型 |
| `source_type` | 主要来源类型 |
| `agent` | 产出或整理该文档的 Agent |
| `round` | 来源采集轮次 |
| `status` | 文档审查状态 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `version` | 文档版本 |
| `path` | 本地保存路径 |
| `sources` | 来源列表 |

### 5.2 KnowledgeForge contract

contract 块位于正文后、变更记录前，使用 HTML 注释包裹：

```markdown
<!-- knowledgeforge:contract
{ ... JSON ... }
-->
```

contract 字段说明：

| 字段 | 说明 |
|---|---|
| `file_id` | 与 front matter `id` 对齐的稳定文件 ID |
| `file_path` | 实际保存路径 |
| `required_sections` | 该文件必须具备的章节 |
| `claims` | 文件中需要证据支撑的核心结论 |
| `evidence_needed` | 需要补充的证据类型 |
| `query_tasks` | 文件级证据链接任务列表 |
| `completion_status` | 文件完成状态与 task id 统计 |

`query_tasks` 最小字段：

| 字段 | 说明 |
|---|---|
| `task_id` | 领域队列中的任务 ID，文件内唯一 |
| `task_type` | 固定为 `query` |
| `section` | 关联章节，通常是 `证据与来源` |
| `claim_or_gap` | 需要补证据的结论或缺口 |
| `query_text` | 证据查询语句 |
| `expected_evidence` | 期望链接类型 |
| `preferred_source_types` | 首选来源类型 |
| `acceptance_criteria` | 任务完成判定 |
| `status` | `pending|running|completed|insufficient|failed` |
| `citation` | 链接任务完成后的来源信息 |
| `selected_link` | 最终选中的官方或高公信力链接 |
| `source_kind` | `official_documentation` / `standard` / `paper` / `project_homepage` / `wikipedia` 等 |
| `reachable` | 链接是否可访问 |
| `relevance_reason` | 为什么该链接最贴近目标知识点 |
| `checked_at` | 链接验证时间 |

`completion_status.state` 推荐值：

- `planned`
- `generated`
- `link_querying`
- `link_verified`
- `link_failed`
- `partially_completed`
- `completed`
- `insufficient`
- `failed`

链接记录要求：

- 每个链接任务完成后，必须更新 `knowledge_task_queue.json` 中该任务的 `status`、`citation`、`selected_link`、`source_kind`、`reachable`、`relevance_reason` 和 `checked_at`。
- 主链路不抓取网页内容补正文，不追加 Agent 贡献区，不生成内容 summary。
- contract 保留链接任务定义，完整文档补全阶段可消费队列中的可信链接继续生成正文。

### 5.3 推荐枚举

`doc_type` 推荐值：

- `source`：原始资料整理
- `article`：结构化知识文章
- `summary`：摘要或综述
- `case`：案例
- `trend`：趋势和观点
- `note`：补充说明

`source_type` 推荐值：

- `insight`：本地知识或历史沉淀
- `query`：外部事实、官方或权威来源
- `media`：媒体、社区或热点观点
- `manual`：人工录入
- `mixed`：多来源融合

`status` 推荐值：

- `draft`：初始保存，尚未质量校验
- `reviewed`：已完成基础审查
- `verified`：已通过质量检测，可进入稳定版本
- `deprecated`：已废弃，不再作为有效知识使用

## 6. 写作要求

1. 标题必须具体，不使用“资料整理”“总结”这类泛标题。
2. 摘要应先讲结论，再讲范围，避免只描述写作过程。
3. 正文使用二级、三级标题组织，不跳级使用标题。
4. 每个关键结论都应能在“证据与来源”中找到支撑。
5. 原始引用应控制长度，优先转述和注明来源。
6. 冲突、不确定性和缺口必须显式记录，不能隐藏在正文里。
7. 后续动作必须可执行，例如“补充 2025 年后官方材料”，而不是“继续研究”。
8. 表格只用于结构化信息，长篇解释仍放在正文段落中。

## 7. 与 Neo4j 的映射建议

统一 Markdown 文档应支持以下图谱映射：

| Markdown 字段或章节 | Neo4j 映射 |
|---|---|
| `domain` | `Domain` 节点 |
| `subdomain` | `SubTopic` 节点 |
| `id`、`title`、`path` | `Article` 节点属性 |
| `sources` | `Source` 节点或来源属性 |
| 实体候选 | `Concept`、`Tool`、`Organization` 等实体节点 |
| 关系候选 | 实体之间的关系边 |
| `status`、`version` | 审查状态与版本属性 |
| contract `completion_status.state` | `KnowledgeStructureNode.generation_state` |
| contract `completed_task_ids` | `KnowledgeStructureNode.completed_task_count` |
| contract `pending_task_ids` | `KnowledgeStructureNode.pending_task_count` |

路径关联的基本要求：

- `Article.path` 必须与 Markdown front matter 中的 `path` 一致。
- 图谱中的 `Article.id` 必须与 Markdown front matter 中的 `id` 一致。
- 文档状态从 `draft` 变为 `verified` 后，才可作为研报生成的可信输入。
- 知识结构节点的 `generated_path` 必须指向实际 Markdown 文件。
- 文件 contract 完成状态变化后，必须同步更新对应图谱节点。

## 8. 质量检查清单

保存或更新知识文档前，应至少检查：

- 是否为 Markdown 文件。
- 是否包含完整 YAML front matter。
- `domain`、`subdomain`、`path` 是否与实际目录一致。
- 是否至少包含一个来源；Insight 类历史资料也应指向本地来源或历史记录。
- `framework` 模式下，知识定位、学习角色与路径、知识关系、证据与来源是否完整。
- `full_document` 模式下，摘要、关键结论、正文、证据与来源是否完整。
- 关键结论是否有证据支撑。
- 实体与关系候选是否保留来源指针。
- 冲突与不确定性是否显式记录。
- 变更记录是否更新。
- contract 是否存在且 JSON 可解析。
- `query_tasks`、`completed_task_ids`、`pending_task_ids` 是否与领域队列一致。
