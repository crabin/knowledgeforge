# KnowledgeForge 知识文档格式规范

> 目的：为所有需要保存到本地知识库的知识文档提供统一、通用、易读的 Markdown 存储结构，确保文档既适合人工阅读，也适合后续结构化抽取、Neo4j 路径关联、质量检测和版本管理。

## 1. 适用范围

本规范适用于 KnowledgeForge 中所有进入知识沉淀链路的 Markdown 文件，包括：

- 原始资料整理文档
- Agent 中间产物文档
- 结构化后的知识文章
- 子领域补充材料
- 观点、趋势、案例类汇总文档

领域结构目录索引文档和版本记录文档可以使用专门模板，但仍应遵守基础元信息、来源追溯和路径稳定性要求。

---

## 2. 文件命名与保存位置

### 2.1 保存位置

知识文档按以下层级保存：

```text
save/{领域名称}/{子领域名称}/{文档文件名}.md
```

领域目录下必须包含结构目录索引文档：

```text
save/{领域名称}/README.md
```

### 2.2 文件命名

推荐文件名格式：

```text
{YYYYMMDD}-{短标题}-{来源类型}.md
```

示例：

```text
20260423-langgraph-state-management-query.md
20260423-agent-workflow-practices-media.md
20260423-local-knowledge-overview-insight.md
```

命名要求：

- 使用小写英文、数字和短横线，减少跨系统路径兼容问题。
- 文件名应稳定，不应因为标题轻微修改频繁变化。
- 同一子领域内文件名不得重复。
- 文件移动或重命名后，必须同步更新 Neo4j 路径关联和版本记录。

---

## 3. 统一文档结构

每篇知识文档必须使用以下结构：

```markdown
---
id: "{stable_document_id}"
title: "{文档标题}"
domain: "{领域名称}"
subdomain: "{子领域名称}"
doc_type: "source|article|summary|case|trend|note"
source_type: "insight|query|media|manual|mixed"
agent: "{agent_name}"
round: 1
status: "draft|reviewed|verified|deprecated"
created_at: "YYYY-MM-DDTHH:mm:ss+09:00"
updated_at: "YYYY-MM-DDTHH:mm:ss+09:00"
version: "v1"
path: "save/{领域名称}/{子领域名称}/{文档文件名}.md"
tags:
  - "{tag}"
sources:
  - title: "{来源标题}"
    url: "{来源链接或本地路径}"
    publisher: "{发布方}"
    retrieved_at: "YYYY-MM-DDTHH:mm:ss+09:00"
    reliability: "high|medium|low|unknown"
---

# {文档标题}

## 摘要

用 3 到 6 句话说明本文档沉淀的核心内容、适用范围和主要结论。

## 关键结论

- 结论 1。
- 结论 2。
- 结论 3。

## 背景与上下文

说明该知识出现的领域背景、问题边界、时间范围、适用场景和必要前提。

## 正文

### {一级主题}

围绕一个清晰主题展开，优先使用短段落、列表和小标题。

### {一级主题}

继续补充事实、观点、案例或机制说明。

## 证据与来源

| 编号 | 来源 | 关键信息 | 可信度 | 备注 |
|---|---|---|---|---|
| S1 | {来源标题或链接} | {支持的结论或事实} | high | {说明} |

## 实体与关系候选

### 实体候选

| 实体 | 类型 | 描述 | 来源 |
|---|---|---|---|
| {实体名} | {Domain/SubTopic/Concept/Tool/Organization/Person/Event} | {简述} | S1 |

### 关系候选

| 主体 | 关系 | 客体 | 来源 |
|---|---|---|---|
| {实体 A} | {关系类型} | {实体 B} | S1 |

## 冲突与不确定性

- {尚未裁决的冲突、口径差异或证据不足点。没有则写“暂无”。}

## 后续动作

- {需要补检索、复核、结构化修复或人工确认的事项。没有则写“暂无”。}

## 变更记录

| 版本 | 时间 | 变更说明 |
|---|---|---|
| v1 | YYYY-MM-DD | 初始创建 |
```

## 4. 字段说明

### 4.1 必填元信息

| 字段 | 说明 |
|---|---|
| `id` | 稳定文档 ID，用于去重、版本追踪和图谱映射 |
| `title` | 文档标题，应清晰表达主题 |
| `domain` | 所属领域 |
| `subdomain` | 所属子领域 |
| `doc_type` | 文档类型 |
| `source_type` | 主要来源类型 |
| `agent` | 产出或整理该文档的 Agent |
| `round` | 来源采集轮次 |
| `status` | 文档审查状态 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `version` | 文档版本 |
| `path` | 本地保存路径 |
| `sources` | 来源列表 |

### 4.2 推荐枚举

`doc_type` 推荐值：

- `source`：原始资料整理
- `article`：结构化知识文章
- `summary`：摘要或综述
- `case`：案例
- `trend`：趋势和观点
- `note`：补充说明

`source_type` 推荐值：

- `insight`：本地知识或历史沉淀
- `query`：外部事实、官方或权威来源
- `media`：媒体、社区或热点观点
- `manual`：人工录入
- `mixed`：多来源融合

`status` 推荐值：

- `draft`：初始保存，尚未质量校验
- `reviewed`：已完成基础审查
- `verified`：已通过质量检测，可进入稳定版本
- `deprecated`：已废弃，不再作为有效知识使用

---

## 5. 写作要求

1. 标题必须具体，不使用“资料整理”“总结”这类泛标题。
2. 摘要应先讲结论，再讲范围，避免只描述写作过程。
3. 正文使用二级、三级标题组织，不跳级使用标题。
4. 每个关键结论都应能在“证据与来源”中找到支撑。
5. 原始引用应控制长度，优先转述和注明来源。
6. 冲突、不确定性和缺口必须显式记录，不能隐藏在正文里。
7. 后续动作必须可执行，例如“补充 2025 年后官方材料”，而不是“继续研究”。
8. 表格只用于结构化信息，长篇解释仍放在正文段落中。

---

## 6. 与 Neo4j 的映射建议

统一 Markdown 文档应支持以下图谱映射：

| Markdown 字段或章节 | Neo4j 映射 |
|---|---|
| `domain` | `Domain` 节点 |
| `subdomain` | `SubTopic` 节点 |
| `id`、`title`、`path` | `Article` 节点属性 |
| `sources` | `Source` 节点或来源属性 |
| 实体候选 | `Concept`、`Tool`、`Organization` 等实体节点 |
| 关系候选 | 实体之间的关系边 |
| `status`、`version` | 审查状态与版本属性 |

路径关联的基本要求：

- `Article.path` 必须与 Markdown front matter 中的 `path` 一致。
- 图谱中的 `Article.id` 必须与 Markdown front matter 中的 `id` 一致。
- 文档状态从 `draft` 变为 `verified` 后，才可作为研报生成的可信输入。

---

## 7. 质量检查清单

保存或更新知识文档前，应至少检查：

- 是否为 Markdown 文件。
- 是否包含完整 YAML front matter。
- `domain`、`subdomain`、`path` 是否与实际目录一致。
- 是否至少包含一个来源；Insight 类历史资料也应指向本地来源或历史记录。
- 摘要、关键结论、正文、证据与来源是否完整。
- 关键结论是否有证据支撑。
- 实体与关系候选是否保留来源指针。
- 冲突与不确定性是否显式记录。
- 变更记录是否更新。

---
